# UI 质量检查清单

本文档用于 UI 任务的实现前评审和交付前验收。视觉方向见 `docs/ui-guidelines.md`，shadcn 维护规则见 `docs/shadcn-maintenance.md`，Nuxt 边界见 `NUXT_AI_CONSTRAINTS.md`。

本清单按本仓库定位裁剪：这里是 C 端 Nuxt starter，不是后台管理系统，也不是移动原生应用。

## 使用时机

涉及以下任一内容时，至少过一遍相关小节：

- 新页面、首页、产品页、登录页、空状态或错误状态。
- 新增或重构表单、弹窗、菜单、导航、卡片、列表、图表。
- 调整配色、字体、间距、动效、响应式布局。
- 修复移动端溢出、遮挡、点击困难、状态不清晰等体验问题。
- 上线前做 UI、可访问性或真实渲染检查。

纯后端逻辑、脚本、基础设施、API 数据结构变更不需要套用本清单。

## 优先级

按下面顺序处理问题。高优先级问题没有解决时，不要用低优先级装饰掩盖。

| 优先级 | 类别 | 必须检查 | 常见反模式 |
| --- | --- | --- | --- |
| P0 | 可访问性 | 对比度、focus ring、键盘可达、图标按钮标签 | 移除 outline、只用颜色表达状态 |
| P0 | 交互反馈 | loading、disabled、错误、成功、重复提交 | 点击后无反馈、异步中还能重复点 |
| P1 | 移动端布局 | 375px 可读、无横向滚动、固定元素不遮挡 | 固定宽度、按钮文字溢出、sticky 覆盖内容 |
| P1 | Nuxt 边界 | 页面保持数据编排，交互下沉到 `app/features` | 客户端组件直接导入 `server/*` |
| P1 | 主题和组件 | 使用语义 token、common/ui 组件、lucide 图标 | 业务组件散落硬编码颜色和自造基础控件 |
| P2 | 动效 | 150-300ms、尊重 reduced motion、只表达状态变化 | 装饰性循环动效、动画导致 layout shift |
| P2 | 性能 | 图片尺寸、LCP 图、字体加载、懒加载 | 无尺寸图片、首屏加载重组件、布局跳动 |
| P3 | 图表和数据 | 标题、单位、图例、空/错/加载状态 | 只靠颜色区分、移动端轴标签挤压 |

## 实现前检查

开始写 UI 代码前，先明确这些点：

- 页面核心任务是什么，首屏应该让用户完成或理解什么。
- 主要 CTA 只有一个，次要操作是否明显降级。
- 首屏和 SEO 内容是否能由 Nuxt SSR 直接渲染。
- 交互边界是否足够小，是否只在必要组件里使用浏览器状态。
- 数据是否通过 Nitro API 或 `server/queries/*` 读取，页面是否避免直接导入 `server/mocks/*`。
- 能否复用 `app/components/common` 或 `app/components/ui`，而不是新增一套基础组件。
- 是否需要表单校验、loading、错误、空状态、成功反馈和防重复提交。
- 是否触发 ADR 条件，例如新增依赖、认证、远程数据源、缓存策略或部署行为。

## 视觉和布局

- 页面保持 C 端、轻量、清晰，不做后台仪表盘式密集界面。
- 使用移动优先布局，至少检查 375px、768px、1024px、1440px。
- 正文行宽控制在可读范围，长文本不要在桌面端横跨整屏。
- 按钮、卡片、表单控件不能因文案变长而溢出或遮挡。
- 固定导航、底部 CTA、toast、dialog、popover 必须留出内容避让空间。
- 避免卡片套卡片；页面区块优先用自然分区，不把整页做成浮动卡片集合。
- Glassmorphism 只用于少量悬浮层级，不承载长正文、主表单或核心信息。
- Mesh gradient 只做氛围层，文字可读性不能依赖背景刚好合适。

## 主题和组件

- 业务组件使用 `background`、`foreground`、`primary`、`muted`、`border`、`ring`、`destructive` 等语义 token。
- 不在页面里散落固定品牌色；确需扩展时优先新增语义变量或 common 包装组件。
- 图标优先使用 `@lucide/vue`，同一区域保持一致的尺寸、线宽和视觉风格。
- 图标按钮必须有可访问名称，例如 `aria-label` 或可见文本。
- 基础 shadcn-vue 源码只放在 `app/components/ui`，业务包装放 `app/components/common`。
- 表单优先使用 vee-validate + Zod，并使用已有 Form/Input/Button 模式。
- Dialog、Dropdown、Popover 等浮层必须有清晰标题、关闭路径和焦点管理。

## 可访问性

- 普通正文对比度目标不低于 4.5:1，大字号和图形元素不低于 3:1。
- 所有可交互元素必须有可见 focus 状态，不能只保留 hover 状态。
- 键盘 Tab 顺序应匹配视觉顺序，弹窗打开后焦点进入弹窗，关闭后返回触发元素。
- 图片有意义时必须有 `alt`；纯装饰图片使用空 `alt`。
- 表单字段必须有可见 label，不用 placeholder 代替 label。
- 错误信息靠近对应字段，并说明原因和修复方式。
- 不只用颜色表达成功、错误、选中、禁用等状态，应配合文字、图标或边框。
- 动效必须尊重 `prefers-reduced-motion`。

## 交互反馈

- 异步按钮提交后进入 loading/disabled 状态，防止重复提交。
- 用户操作成功后给出短反馈，例如 toast、状态文案或按钮状态变化。
- 失败状态显示原因和下一步，例如重试、修改输入、返回上一页。
- 删除、退出、覆盖等破坏性操作需要确认或撤销路径。
- 点击区域在移动端至少接近 44px 高宽，图标按钮要通过 padding 扩大命中范围。
- hover 不能作为唯一交互入口；触屏用户也必须能完成同样操作。
- 空状态要说明当前为什么为空，并给出合理下一步。

## 动效

- 动效服务于层级、状态变化或空间关系，不做纯装饰堆叠。
- 常规微交互控制在 150-300ms，复杂转场不超过 400ms。
- 优先动画 `transform` 和 `opacity`，避免动画 `width`、`height`、`top`、`left`。
- 进入动效可稍慢，退出动效应更快，避免阻塞用户下一步操作。
- 骨架屏或占位内容要预留尺寸，不能造成内容加载后跳动。

## 性能和稳定性

- 首屏关键图片设置明确尺寸、比例和合理加载策略。
- 非首屏图片和重组件懒加载；不要把图表、编辑器、地图等重组件放进首屏主 bundle。
- 字体加载应避免 FOIT 和布局抖动。
- 服务端传给客户端交互组件的 props 只包含客户端实际需要的字段。
- 独立异步读取尽量并行，避免无意义 waterfall。
- 列表超过几十项时关注虚拟化、分页或服务端过滤。
- 不要用任意大的 `z-index` 解决遮挡，应建立清晰层级。

## 图表和数据

只有在产品真的需要数据表达时才引入图表。营销页或简单状态页优先用清晰文案和少量指标。

- 趋势用折线，比较用柱状，构成比例少于 5 类时才考虑饼图或环图。
- 图表必须有标题、单位、时间范围和必要图例。
- 不只靠红绿区分数据；为色弱用户提供形状、纹理、标签或直接数值。
- 移动端图表要简化刻度和标签，避免轴文字旋转堆叠。
- 图表数据加载、为空、失败时都有明确状态。
- 复杂数据提供表格或文字摘要，不能让图表成为唯一信息来源。

## 交付前验证

UI 变更完成前至少检查：

- 小屏 375px 无横向滚动，文字、按钮、卡片不溢出。
- 桌面宽屏下正文不铺满，页面层级仍清楚。
- 键盘可操作主要交互，focus ring 可见。
- loading、empty、error、success、disabled 状态都能被触发或合理说明。
- light/dark 主题下文本、边框、禁用态、错误态都可辨认。
- `prefers-reduced-motion` 下没有强制大幅动画。
- 固定导航、底部操作栏、toast、dialog 不遮挡主要内容。
- 浏览器控制台没有由本次变更引入的错误。

涉及 UI 的代码改动优先用 Playwright 或浏览器截图验证真实渲染；纯文档变更按 `docs/codex-playbook.md` 的文档验证命令检查引用。
