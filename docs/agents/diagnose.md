# 诊断流程

本文档用于难复现 bug、测试失败、运行时报错和性能退化。目标是先建立可靠反馈环，再定位根因；不要靠猜测式修复推进。

## 触发场景

- 用户报告“坏了”“失败”“报错”“很慢”。
- CI、单元测试、端到端测试或构建失败。
- 行为和预期不一致，但根因尚不明确。
- 性能退化、偶发失败或竞态问题。

## 流程

1. 建立反馈环
   - 优先写能失败的测试。
   - 其次使用 `curl`、脚本、Playwright、fixture 或最小复现。
   - 偶发问题要提高复现率，例如循环运行、固定时间、固定随机种子或缩小并发窗口。
   - 如果无法建立反馈环，明确列出已尝试的方法，并向用户索要日志、HAR、录屏、环境访问或临时插桩许可。

2. 复现问题
   - 运行反馈环，确认失败现象和用户描述一致。
   - 记录具体报错、错误输出、慢查询或错误 UI 状态。
   - 不要把相邻但不同的问题当成原问题修。

3. 提出可证伪假设
   - 给出 3 到 5 个按可能性排序的假设。
   - 每个假设必须写清预测：如果原因是 X，那么改变或观察 Y 会出现什么结果。
   - 如果用户在线，先展示假设列表；用户不在线时按最高可能性继续。

4. 定向验证
   - 一次只改变一个变量。
   - 优先用调试器、REPL、最小脚本或精准日志。
   - 临时日志必须使用唯一前缀，例如 `[DEBUG-a4f2]`，结束前用 `rg` 清理。
   - 性能问题先建立基准，再比较，不要先改代码。

5. 修复和回归测试
   - 有合适测试边界时，先把最小复现固化为失败测试，再修复。
   - 如果没有合适测试边界，记录这个架构风险，不要写虚假的浅层测试。
   - 修复后重跑最小复现和原始复现场景。

6. 收尾
   - 原始问题不再复现。
   - 回归测试通过，或已说明无法自动化的原因。
   - 临时日志、脚本和原型已删除，或移动到明确标记的调试位置。
   - 最终说明根因、修复方式、验证命令和剩余风险。

## 本仓库约束

- 运行 Node 相关命令前必须先加载 nvm：

  ```bash
  export NVM_DIR="$HOME/.nvm" && . "$NVM_DIR/nvm.sh"
  nvm use
  ```

- 涉及 Nuxt、Vue、Nitro、runtime config、SSR 或安全边界时，先读 `NUXT_AI_CONSTRAINTS.md`。
- 页面不得直接导入 `server/mocks/*`，必须通过 Nitro API 或 `server/queries/*`。
- 客户端模块不得导入 `server/*`。
- 修复完成后按影响范围运行 `docs/codex-playbook.md` 中的验证矩阵。
