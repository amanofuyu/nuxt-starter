# 原型验证流程

本文档用于在正式实现前快速回答一个不确定问题。原型是可丢弃代码，不是半成品实现。

## 触发场景

- 状态机、数据形状或业务流程仅靠文档难以判断。
- UI 信息层级、布局方向或交互方式需要比较。
- 用户明确要求“做个原型”“试几个方案”“让我玩一下”。
- 需要用低成本实验降低正式实现风险。

## 先定义问题

原型必须只回答一个问题。开始前写清：

- 要验证的假设是什么。
- 用户或维护者如何体验原型。
- 什么结果说明方向可行。
- 原型结束后哪些代码会删除，哪些经验会进入正式实现。

## 两种原型

### 逻辑原型

适合状态流转、数据结构、业务规则和复杂边界。

- 使用最小 CLI、脚本或测试夹具驱动真实规则。
- 默认使用内存状态，不连接真实数据库或外部服务。
- 逻辑模块和命令行交互分离，避免把 `console`、prompt 或终端控制码写进核心逻辑。
- 原型不需要完整测试；如果已经需要长期测试，说明它不再是原型，应转为正式实现任务。

### UI 原型

适合页面布局、信息架构、表单流程和交互比较。

- 在临时路由或隔离组件中提供 2 到 3 个结构明显不同的方案。
- 差异应体现在信息层级、布局、主操作和交互路径，而不只是颜色变化。
- 移动端和桌面端都要能看。
- 用户选定方向后，删除原型切换器和未采用方案，再进入正式实现。

## 约束

- 不引入生产依赖，除非先按 `AGENTS.md` 走联网查询和人工确认。
- 不改认证、权限、缓存、部署或环境变量边界。
- 不把原型接入真实用户数据。
- 不把原型代码混入 `app/components/ui`。
- 不让原型绕过 `NUXT_AI_CONSTRAINTS.md` 的 server/client 边界。

## 收尾

原型完成后必须说明：

- 验证了什么问题。
- 观察到什么结论。
- 哪些代码保留、删除或转为正式实现。
- 是否需要 ADR；如果只是一次性探索且不产生长期决策，说明不需要。
