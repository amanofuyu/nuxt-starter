# TDD 流程

本文档用于新功能、bugfix、行为变更和重构。核心原则：测试验证可观察行为和公共接口，不绑定实现细节。

## 触发场景

- 新增用户可见能力。
- 修复 bug 或锁定回归。
- 改变 API、组件、状态流转或数据读取行为。
- 重构共享逻辑、服务端查询、API route 或客户端交互。

纯文档、纯样式、一次性原型和不改变行为的配置更新可以不强制 TDD，但最终仍要有合适验证方式。

## 流程

1. 识别行为
   - 写清用户或调用方能观察到什么结果。
   - 明确不变量、错误路径和边界条件。
   - 优先选择最高层、最稳定的测试边界。

2. 红灯
   - 一次只写一个失败测试。
   - 测试名称描述行为，不描述实现。
   - 运行测试并确认它因目标行为缺失而失败，而不是因为拼写、导入或环境错误。

3. 绿灯
   - 写最少实现让当前测试通过。
   - 不提前实现未来需求。
   - 不在红灯状态下重构。

4. 重构
   - 全部相关测试通过后再整理命名、重复和边界。
   - 重构只改变结构，不改变行为。
   - 每次重构后重新运行相关测试。

5. 重复
   - 为下一个重要行为写下一个失败测试。
   - 不一次性写完所有测试再实现。

## 测试取舍

- 优先测试公共接口、页面行为、API 响应、server query 或用户交互。
- 不测试私有函数、内部调用顺序和纯实现细节。
- 不通过删除测试、放宽关键断言或静默吞错让测试通过。
- mock 只用于不可控边界，例如网络、时间、随机数或外部服务；能用本地真实实现时优先真实实现。

## 本仓库建议边界

- 页面数据：优先通过 Nitro API、`server/queries/*` 或 E2E 页面行为测试。
- API route：优先测试请求、响应、schema 和错误码。
- 客户端交互组件：优先测试用户交互和可见状态。
- AI 边界：使用 `tests/unit/ai-boundaries.test.ts` 和 `scripts/check-ai-boundaries.mjs` 保护导入规则。

## 验证

运行 Node 相关命令前必须先加载 nvm：

```bash
export NVM_DIR="$HOME/.nvm" && . "$NVM_DIR/nvm.sh"
nvm use
```

普通代码改动至少运行：

```bash
corepack pnpm@11.6.0 verify
```

涉及 UI、Docker 或发布时，按 `docs/codex-playbook.md` 追加对应验证。
